연속리스트
	기억저장소 이용 효율은 밀도 1로 가장 좋다

트리 관련 용어
	디그리(차수): 각 노드에서 뻗어나온 가지의 수
		* 한 트리의 차수는 가장 큰 차수를 말한다.
	형제노드: 동일한 부모를 갖는 노드들

데이터베이스
	통합된 데이터: 자료의 중복을 배제한 데이터의 모임
	저장된 데이터: 컴퓨터가 접근할 수 있는 저장 매체에 저장된 자료
	운영 데이터: 업무 수행에 필요한 필수 데이터
	공용 데이터: 여러 응용 시스템들이 공동으로 소유하고 유지하는 자료

DBMS의 필수기능
	정의, 조작, 제어
	
DBMS의 제어기능
	무결성, 권한 검사, 병행 제어(결과가 항상 정확하게 유지되도록)

스키마
	외부스키마: 각 개인의 입장에서 필요로하는 DB의 논리적 구조를 정의한 것
	개념 스키마: DB의 전체적인 논리적 구조
	내부 스키마: DB에 저장될 레코드의 형식

소프트웨어 패키징
	사용자를 중심으로 진행
	암호화 및 보안을 고려
	DRM 연동을 고려


DRM
	패키저(제공자 측): 배포 가능한 형태로 묶어 암호화함
	DRM 컨트롤러(소비자 측): 콘텐츠의 이용 권한을 통제
	
형상관리
	소프트웨어 개발 전 단계에 수행되며 유지보수 단계에도 수행됨
	전체 개발 비용을 줄이고 방해요인 최소화가 목적
	관리 항목에는 소스 코드, 정의서, 지침서, 분석서 등이 포함된다
	소프트웨어의 진행 정도를 확인하기 위해 사용되기도 함
	
소프트웨어 버전등록 과정
	Import - CheckOut - commit - Update - Diff
	* CheckIn, Repository도 주요기능에 있다

버전관리도구
	클라이언트/서버: 서버가 문제가 되면 개발을 못 함
	분산저장소: 로컬에서도 버전 관리 가능하므로 원격 저장소에 문제가 생겨도 괜찮음
	서브비전(SVN): 클라이언트/서버 방식으로 Trunk 에서 작업 후 Commit 할 때마다 리비전이 1씩 증가
	GIT: 분산저장소 방식으로 파일의 변화를 스냅샷으로 저장함

테스트
	테스트는 작은부분 -> 큰 부분으로 진행
	개발자와 관계없는 팀이 진행
	살충제 패러독스: 동일한 테스트 케이스로 반복하면 결함이 있어도 잡아내질 못 함
	오류-부재의 궤변: 소프트웨어 결함을 모두 제거해도 사용자의 요구사항을 만족시키지 못 한것

회귀 테스트: 수정된 코드에 결함이 없음을 확인
병행 테스트: 수정 전, 후 코드에 input에 따른 출력을 비교

V-모델
	단위 테스트: 각 모듈 테스트
	통합 테스트: 모듈 결합 후 테스트
	시스템 테스트: 시스템에 배포 후 테스트
	인수 테스트: 요구사항 충족하는지, 알파, 베타 테스트

하향식 통합 테스트
	드라이버보다 작성하기 쉬움
	가짜 모듈 사용

순환 복잡도
	선으로 만들어진 각 영역의 수를 세면 됨(화살표 방향 필요없음)
	
소스코드 품질 분석도구
	동적: Avalanche, Valgrind	(AV)
	정적: 위에 것 외에 모두

모듈 연계 방식
	Hub N Spoke: 중앙 집중형
	Message Bus: 앱 사이 미들웨어를 둠
					* 앱이 아닌 서비스 중심의 통합을 지향함
	Hybrid: 위 두것을 합침
	
모듈 세부 설계서
	컴포넌트 명세서: 컴포넌트 관련과 내부 클래스, 인터페이스 클래스에 대해 정의함
	인터페이스 명세서: 인터페이스 클래스를 자세히 정의
	
인터페이스 예외처리
	데이터 통신이용: 인터페이스 "객체" 송, 수신시 이용
		* 시스템 환경, 송-수신 데이터, 프로그램 자체원인에 사용
	인터페이스 엔티티 이용: 인터페이스 "테이블" 송, 수신시 이용
		* 위에 원인 외에 전부(일반적으로 인터페이스 관련임)

인터페이스 구현 검증 도구
	STAF: 분산환경, 컴포넌트 재사용, 데몬 등이 키워드
	FitNesse: 웹 기반 테스트케이스 설계, 실행 도구
	NTAF: 위 두개를 통합한 네이버의 프레임워크
	셀러리엄: 다양한 브라우저 지원
	watir: Ruby 사용
	
인터페이스 감시도구
	APM을 사용: 스카우터, 제니퍼 등있음
